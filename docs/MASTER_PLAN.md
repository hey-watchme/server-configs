## 1. プロダクトの位置づけ（2026-07-05 確定）

WatchMe 本体（iOS アプリによる感情・環境・発話の分析）のゴールは、**プレゼンテーション用モックアップ**である。

- 目的は、投資家を驚かせ、仲間を集めること。一般ユーザー向けの実用サービスではない
- したがって**コストは度外視してよく、常に最高精度のモデル構成を選ぶ**
- 専用デバイスの開発は資金調達・チームアップ後。それまでは**このアプリで限界までチューニングしてビジョンを表現する**ことが最大の課題
- 音声から感情・気分・特性を高解像度で読み取る基盤技術は自社開発しない。世の中で最良のものをアプリケーションレイヤーで使う

### 入力の3経路（現行）

1. **アプリ内マイク録音** — 「録音→即分析→今の環境・感情・発話が見える」体験の核
2. **ウェアラブル議事録デバイス → Bluetooth → モバイル → ファイル分析** — 長時間モニタリング
3. **カメラロールの動画** — 音声抽出して分析

⚠️ Observer デバイス（ESP32、30分×48回の定点録音）は**凍結済み**。マイク性能とユースケース不在が理由。ドキュメントで Observer 前提の記述を見たら古い情報と判断すること。

## 2. ゴールの定義

以下の2つが達成されれば「あるべき姿」とする。

1. **表現力**: 1本の録音から「どんな環境で、何が起き、誰が何を話し、どんな感情だったか」が、見た人が驚くレベルの解像度と品質で可視化される
2. **堅牢性**: デモの場で絶対に壊れない。録音からダッシュボード表示まで、失敗しても状態が見え、リカバリーできる

## 3. 現在地（2026-07-05、DB実測に基づく）

- 実録音・実分析の最終稼働: **2026-04-23**（約2.5ヶ月凍結）
- **SER は死んでいる**: Hume Expression Measurement API が 2026-06-14 でサンセット（[TECH_INVENTORY_2026-07.md](./TECH_INVENTORY_2026-07.md) §2）。復旧ではなく再設計が必須
- ASR（Speechmatics）/ SED（AST）/ Profiler（gpt-5.4）は凍結時点まで稼働。ただし ASR/SED は最新世代との比較未実施
- **デモデータ生成 Lambda が毎時稼働継続中**（デモデバイスが spot_results に書き込み中）→ 停止/継続の判断が必要
- 既知の構造課題（KNOWN_ISSUES.md）: Lambda→EC2 のタイムアウト、監視の欠如、DLQ 蓄積、エンドポイント不統一
- 根本課題: **14 SQS キュー + 11 Lambda + 3並列パイプライン合流**という構成が、少人数デモ用途に対して過剰に複雑で、壊れやすさの主因

## 4. ターゲットアーキテクチャ（提案）

「3パイプライン並列 + SQS 合流」から、**2系統ハイブリッド**へ簡素化する。

```
録音（iOS: マイク / BTデバイス / 動画）
  │
  ├─ ① 専用ASR: 逐語文字起こし + 話者分離 + 単語タイムスタンプ（計測の骨格）
  │      候補: ElevenLabs Scribe v2 / AssemblyAI Universal-3 Pro / Speechmatics Melia-1
  │
  └─ ② マルチモーダルLLM（音声直接入力）: 感情・トーン・環境音・状況の構造化記述（解釈）
         候補: Gemini 3.1 Pro（第一候補）/ Qwen3.5-Omni（対抗）
  │
  └→ ③ 統合・プロファイリングLLM: ①+②を突合し、心理プロファイル + プレゼン品質のナラティブ生成
         候補: 最新フロンティア（GPT-5.x / Claude / Gemini）
```

- 現行の SED（AST 527クラス）と SER（Hume）は ② に吸収して**リタイア**（固定クラスより自由記述の方が表現力が高く、プレゼン用途に合う）
- グラフ用の一貫スコア（vibe_score 等）が LLM 出力で安定しない場合のみ、emotion2vec+ / SenseVoice 等の専用モデルを補助的に併用
- パイプライン段数が減る = 壊れる箇所が減る。合流待ち（aggregator-checker）が不要になり、SQS/Lambda を大幅削減できる

- [x] 現状把握・技術棚卸し・ドキュメントのピン止め（2026-07-05）

- [ ] AWS/外部サービスの棚卸し（不要リソース・課金の整理）

### Phase 1: 分析コアの刷新（PoC → 選定）

- [ ] 手持ちの実録音（生活雑音入り日本語）でテストセットを作る
- [ ] ASR 3社 A/B（Scribe v2 / Universal-3 Pro / Melia-1）— 日本語逐語精度・話者分離・雑音耐性
- [ ] Gemini 3.1 Pro に同じ音声を直接入力し、感情・環境音・状況記述の品質を検証（vs 旧 AST+Hume 出力）
- [ ] 出力スキーマ設計（vibe_score の互換 or 刷新、感情タクソノミーの再設計）
- [ ] 選定結果でパイプライン実装を差し替え

### Phase 2: パイプラインの簡素化・堅牢化

- [ ] 3並列+合流 → 2系統ハイブリッドへの構成変更（不要な SQS/Lambda の廃止）
- [ ] 処理状態の可視化（アプリから「いまどの段階か・失敗したか」が見える。pipeline_status テーブルの活用）
- [ ] 失敗時のリトライ/再実行手段（デモ中のリカバリー動線）
- [ ] 最低限の監視・アラート（DLQ/エラーの通知）

### Phase 3: プレゼン品質のチューニング（iOS）

- [ ] 新しいリッチな分析出力（環境の自由記述・文脈込み感情）を活かした可視化の再設計
- [ ] 「録音 → 進行表示 → 結果が現れる」というデモの流れの磨き込み
- [ ] 既知バグ潰し（深夜日付ロールオーバー、decode 失敗で当日表示全滅、認証トークン等）
- [ ] デモシナリオ（投資家向けに見せる録音・ストーリー）の整備
